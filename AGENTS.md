# AGENTS.md

## 목적

이 저장소의 문서는 학습 내용을 빠르게 적는 초안이 아니라,
나중에 다시 읽었을 때 스스로 설명할 수 있는 수준의 정리 문서를 목표로 한다.

특히 `notes/` 아래 문서는 개념을 과하게 압축하기보다,
"이게 무엇인지", "왜 필요한지", "어디까지 이해하면 되는지"가 드러나도록 작성한다.

## notes 문서의 실제 공통 패턴

현재 `notes/java`, `notes/architecture` 문서들은 완전히 동일한 템플릿을 쓰지는 않지만,
대체로 아래 구조를 따른다.

1. `# 제목`
2. `## 개요`
3. 주제별 본문 섹션 여러 개
4. `## 핵심 정리`

즉, 엄격한 양식보다는 느슨한 공통 구조가 이미 존재한다고 본다.

## notes 작성 원칙

- 제목은 가능한 한 주제 중심으로 짧게 쓴다.
- 첫 `개요`에서는 문서가 다루는 범위를 명확히 적는다.
- 본문은 강의 요약보다 "내가 다시 설명할 수 있는 문장"으로 정리한다.
- 과한 이론 확장은 피하고, 현재 문서 주제에 필요한 범위까지만 적는다.
- 마지막에는 `핵심 정리`로 다시 압축한다.

## 템플릿 사용 원칙

기본 템플릿은 [notes/_templates/note-template.md](E:/Project/engineering-casebook/notes/_templates/note-template.md)를 따른다.

다만 이 템플릿은 강제 규칙이 아니다.
주제에 따라 아래는 자유롭게 조정할 수 있다.

- 본문 섹션 수
- 예제 유무
- 다이어그램 유무
- "왜 필요한가", "한계", "주의점" 같은 섹션 이름

즉, 일관성은 유지하되 문서를 템플릿에 억지로 끼워 맞추지는 않는다.

## 템플릿을 벗어나도 되는 경우

다음과 같은 경우에는 템플릿을 일부 변형해도 된다.

- 로드맵처럼 계획 문서인 경우
- 비교 문서라서 표나 항목 대조가 중심인 경우
- 실험 기록이라서 단계와 결과가 더 중요한 경우
- 코드 예제가 핵심이라서 예제 섹션 비중이 큰 경우

핵심은 형식 통일보다는 읽는 사람이 빠르게 이해할 수 있는 구조를 만드는 것이다.
