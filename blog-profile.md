# 블로그 프로필

## 목적

이 문서는 기술 블로그 초안 생성과 글 다듬기 과정에서 따라야 하는 실제 글쓰기 기준을 정리한다.

`blog-operations.md`가 운영 정책을 다루는 문서라면, 이 문서는 글의 톤, 구조, 제목, frontmatter, 카테고리/태그 규칙을 다루는 문서다.

## 블로그 정체성

- 블로그 이름: `라프의 실험일지`
- 성격: 학습한 내용을 직접 확인하고, 코드와 구조 관점에서 다시 설명하는 기술 기록
- 중심 주제: 백엔드, 데이터베이스, 아키텍처, 컴퓨터 사이언스, 운영 관점의 정리
- 기본 태도: 배운 내용을 과장해서 소비하지 않고, 이해 가능한 흐름으로 정리한다

## 대상 독자

- 완전 초심자보다는 기본적인 개발 경험이 있는 독자
- 개념을 외우기보다 구조와 동작 원리를 이해하고 싶은 독자
- 실무에서 기술 선택, 설계, 운영 판단에 연결하고 싶은 독자

즉, 입문서식 설명만 반복하기보다 "왜 이 구조를 봐야 하는지", "어떻게 연결되는지"를 알고 싶은 독자를 상정한다.

## 글의 기본 톤

- 설명형 문체를 사용한다.
- 아는 척하는 문장보다 이해를 돕는 문장을 우선한다.
- 과장, 자극, 선언형 마케팅 문구를 피한다.
- 감상문처럼 흐르기보다 구조를 따라 설명한다.
- "정리한다", "본다", "이어서 설명한다", "핵심은 ...이다" 같은 방식의 차분한 기술 설명 톤을 유지한다.

## 글쓰기 원칙

- 개념을 단순 정의로 끝내지 않고 왜 필요한지까지 연결한다.
- 내부 구조나 실행 흐름이 있으면 가능하면 단계적으로 설명한다.
- 실무적 의미가 있으면 운영, 설계, 장애, 성능 관점까지 이어준다.
- 필요 이상으로 이론을 확장하지 않는다.
- 한 글에서 다루는 범위를 분명히 제한한다.
- 요약은 짧게 던지고, 본문에서 그 근거를 풀어낸다.

## 피해야 하는 톤

- 과장된 문제 제기
- 불필요하게 공격적인 단정
- 의미 없이 감성적인 도입
- 이해보다 멋을 우선하는 비유
- 근거 없는 실무 만능론
- 문장마다 결론을 크게 선언하는 스타일

## 기본 글 구조

기본 구조는 아래 흐름을 따른다.

1. 도입
2. 왜 이 주제를 봐야 하는가
3. 핵심 구조 또는 핵심 개념 설명
4. 흐름, 예제, 비교, 문제 해결 과정
5. 운영 또는 설계 관점 정리
6. 마무리

항상 모든 글이 이 구조를 완전히 따를 필요는 없지만, 최소한 아래 두 가지는 유지한다.

- 독자가 왜 이 글을 읽어야 하는지 초반에 드러날 것
- 마지막에는 핵심을 다시 닫아줄 것

## 도입부 원칙

- 첫 문단에서 문제의식이나 핵심 긴장을 잡는다.
- 도입은 짧게 시작하되, 글 전체 주제와 직접 연결되어야 한다.
- 배경 설명이 길어져서 본문 진입이 늦어지지 않게 한다.

좋은 도입의 예시는 대체로 아래 패턴에 가깝다.

- "무엇과 무엇을 함께 만족시켜야 하는가"
- "겉으로는 단순해 보이지만 실제로는 무엇이 핵심인가"
- "처음에는 이렇게 보이지만, 구조를 보면 다르게 이해된다"

## 본문 전개 원칙

- 큰 섹션은 `왜`, `구조`, `흐름`, `운영`, `오해`, `트레이드오프` 같은 이름을 우선 고려한다.
- 설명은 가능하면 단계, 리스트, 표, 다이어그램으로 풀어 독해 부담을 줄인다.
- 개념만 나열하지 말고 관계를 보여준다.
- 실제로 헷갈리기 쉬운 지점은 별도 소제목으로 분리한다.

## 마무리 원칙

- 마무리에서는 글 전체를 다시 압축한다.
- 본문에서 설명한 내용을 번호 목록으로 다시 묶어도 된다.
- 다음 글이나 다음 주제로 이어질 수 있으면 자연스럽게 연결한다.
- 마무리는 감상보다 정리 역할이 우선이다.

## 제목 규칙

제목은 크게 두 패턴을 사용한다.

### 시리즈형

- 예: `[MySQL 아키텍처 4] InnoDB 버퍼 풀(Buffer Pool): 메모리 캐시가 성능과 복구를 동시에 책임지는 이유`
- 같은 주제가 여러 편으로 이어질 때 사용한다.
- 시리즈 번호는 주제 흐름이 명확할 때만 붙인다.

### 단편형

- 예: `순환 구조와 중복 코드를 줄이기 위한 리아키텍처링 기록`
- 특정 문제, 경험, 구조 재설계, 단일 개념 정리에 사용한다.

제목 작성 원칙은 아래와 같다.

- 제목만 봐도 글의 중심 주제가 드러나야 한다.
- 검색 가능한 핵심 용어를 숨기지 않는다.
- 지나치게 낚시성 표현을 쓰지 않는다.
- 같은 시리즈라면 제목 규칙을 일관되게 유지한다.

## 설명문 규칙

frontmatter의 `description`은 짧은 요약이 아니라, 글의 축과 관점을 드러내는 문장으로 작성한다.

원칙은 아래와 같다.

- 무엇을 중심으로 설명하는지 드러낼 것
- 왜 중요한지 또는 어떤 관점으로 보는지 드러낼 것
- 문장 길이는 짧되, 추상적 표현만 남기지 말 것

예시 패턴:

- `X의 내부 구조와 Y 흐름을 중심으로, Z가 왜 중요한지 정리한다.`
- `A와 B의 역할 경계를 기준으로, C를 어떻게 이해해야 하는지 설명한다.`

## Frontmatter 규칙

현재 블로그 포스트 원본은 아래 필드를 사용한다.

```md
---
title: "..."
pubDatetime: 2026-03-07T00:00:00.000Z
description: "..."
categories: [database, learning]
tags: [mysql, innodb, buffer-pool]
draft: false
---
```

초안 생성 시에도 이 형식을 기본으로 맞춘다.

### 필드 원칙

- `title`: 최종 독자 기준 제목
- `pubDatetime`: 발행 시점에 맞춰 설정
- `description`: 글의 범위와 관점을 요약
- `categories`: 상위 분류
- `tags`: 세부 기술 키워드
- `draft`: 블로그 소스 저장소로 옮겨졌지만 아직 공개 발행 전이면 `true`, 실제 공개 발행 시 `false`

## 카테고리 규칙

현재 기준으로 자주 쓰는 상위 분류는 아래와 같다.

- `database`
- `software-engineering`
- `computer-science`
- `learning`

원칙은 다음과 같다.

- 카테고리는 글의 상위 주제를 나타낸다.
- 카테고리는 너무 세분화하지 않는다.
- 대부분의 글에는 `learning`이 함께 붙을 수 있다.

## 태그 규칙

- 태그는 카테고리보다 구체적이어야 한다.
- 기술명, 개념명, 구조명, 문제 유형을 태그로 사용한다.
- 검색 시 유효한 키워드를 우선한다.
- 같은 의미의 태그를 여러 방식으로 중복 사용하지 않는다.

예:

- `mysql`
- `innodb`
- `buffer-pool`
- `architecture`
- `event-driven`
- `validation`
- `spring`

## 시리즈 판단 시의 제목 규칙

- 시리즈 여부는 `notes` 단계에서 미리 고정하지 않는다.
- 초안 생성 시점에 시리즈 후보이면 시리즈형 제목안과 단편형 제목안을 함께 고려할 수 있다.
- 기본값은 단편형 제목이다.
- 주제 누적과 연결성이 명확할 때만 시리즈 번호를 붙인다.

## 표현 도구 사용 원칙

- 코드 블록은 실제로 설명에 필요할 때 넣는다.
- 표는 비교 기준이 분명할 때 사용한다.
- Mermaid는 흐름, 구조, 관계를 보여줄 때 적극적으로 사용할 수 있다.
- 이미지는 설명 보조 역할일 때만 사용한다.

## 초안 생성 시 기대 결과

블로그 초안은 아래 상태를 목표로 한다.

- 제목 초안이 있다
- 설명문 초안이 있다
- 카테고리/태그 초안이 있다
- 도입, 본문, 마무리의 큰 구조가 잡혀 있다
- notes 원문을 그대로 복사한 상태가 아니라 블로그 글 형태로 재구성되어 있다
