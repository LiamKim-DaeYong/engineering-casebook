# 블로그 파이프라인

## 목적

이 문서는 `notes/`에 쌓인 학습 기록을 블로그 초안으로 승격하고, 검토 후 발행 대상으로 넘기기까지의 흐름을 정리한다.

`blog-operations.md`가 운영 정책을 다루고, `blog-profile.md`가 글쓰기 기준을 다룬다면, 이 문서는 실제 작업 절차를 다룬다.

## 기본 흐름

전체 흐름은 아래 순서를 따른다.

1. `notes/`에 학습 기록을 쌓는다.
2. 블로그 초안 후보를 고른다.
3. 초안 디렉토리에 블로그 글 초안을 생성한다.
4. 초안을 검토하며 구조와 표현을 다듬는다.
5. 승인된 초안만 실제 발행 대상으로 넘긴다.

핵심은 발행보다 앞 단계인 `초안 생성과 검토`를 분리하는 것이다.

## 원본과 초안의 분리

- `notes/`는 학습 원본이다.
- 블로그 초안은 원본과 분리된 위치에 둔다.
- 원본 문서를 직접 블로그 발행본처럼 덮어쓰기하지 않는다.

이 분리를 유지해야 학습 메모가 블로그 문체에 끌려가지 않고, 반대로 블로그 초안도 원본 메모의 거친 상태를 그대로 안고 가지 않게 된다.

## 초안 저장 위치

초안은 repo 루트 아래 별도 디렉토리에 저장한다.

기본 위치:

- `blog-drafts/`

이 디렉토리는 실제 블로그 소스 저장소가 아니라, 현재 학습 저장소 안에서 블로그 초안을 관리하는 공간으로 사용한다.

### 초안 상태 디렉토리

초안은 상태별로 아래 디렉토리에 나눠 둔다.

- `blog-drafts/_templates/`: 초안과 후보 스캔에 사용하는 템플릿
- `blog-drafts/inbox/`: 생성 직후 초안
- `blog-drafts/review/`: 검토 중인 초안
- `blog-drafts/approved/`: 발행 직전 승인된 초안
- `blog-drafts/published/`: 발행 후 보관하는 초안

상태가 보이도록 디렉토리를 나누는 이유는 초안이 많아졌을 때 파일만으로 현재 위치를 파악하기 어렵기 때문이다.
초안은 단순 문서가 아니라 작업 중간 상태를 가진 산출물이므로, 디렉토리 자체가 상태를 표현하는 편이 관리에 유리하다.

## 초안 파일의 역할

`blog-drafts/` 아래 문서는 아래 역할을 가진다.

- 블로그 발행용으로 재구성된 초안
- 제목, 설명문, 카테고리, 태그가 포함된 초안
- 시리즈 여부를 검토할 수 있는 중간 산출물
- 사람 검토 이전의 임시 발행 후보

즉, 초안은 메모도 아니고 최종 발행본도 아닌 중간 산출물이다.

## 발행 후 초안 처리 원칙

- 발행이 끝난 초안은 기본적으로 삭제하지 않는다.
- 발행된 초안은 `blog-drafts/published/`로 이동해 보관한다.
- 초안 보관본은 notes 원본과 실제 발행본 사이의 중간 변환 기록으로 본다.

이 보관본은 아래 목적에 유용하다.

- 나중에 글 구조와 톤을 회고할 때 참고
- 하네스 초안 품질을 개선할 때 비교 기준으로 사용
- 동일 주제의 후속 글을 쓸 때 표현과 구조를 재활용

다만 장기적으로 너무 많이 쌓이면, 연도나 주제 기준으로 정리하거나 필요 없는 초안만 선택적으로 수동 삭제할 수 있다.
기본 방침은 삭제보다 보관이다.

## 초안 생성 시점

아래 조건 중 여러 개를 만족하면 초안 생성 후보로 본다.

- 문서의 범위가 어느 정도 닫혔다.
- 개요와 핵심 정리가 채워져 있다.
- 핵심 개념이나 흐름을 내 문장으로 설명할 수 있다.
- 실무, 운영, 설계, 성능, 구조 관점 중 하나 이상으로 확장 가능하다.
- 나중에 다시 읽어도 맥락 없이 이해할 수 있다.

반대로 아래 상태라면 아직 초안 생성보다 학습 기록 보강이 먼저다.

- 내용이 참고 자료 발췌 수준이다.
- 문서의 중심 질문이 아직 정리되지 않았다.
- 예제나 구조 설명 없이 키워드만 모여 있다.

## 후보 탐색 규칙

초안 후보를 찾을 때는 모든 문서를 매번 처음부터 끝까지 다시 읽는 방식을 기본으로 삼지 않는다.
문서가 누적될수록 이 방식은 비용이 너무 크기 때문이다.

후보 탐색은 아래 순서로 좁힌다.

1. 디렉토리 단위로 본다.
2. 파일명과 최근 수정 시점을 본다.
3. 같은 접두사나 같은 주제 축으로 묶는다.
4. 최근 수정되었거나 새로 추가된 문서를 우선 읽는다.
5. 필요할 때만 관련 문서를 추가로 펼쳐 읽는다.

즉, 기본 전략은 `전수 재독`이 아니라 `점진적 후보 축소`다.

후보를 스캔한 결과는 필요하면 `blog-drafts/_templates/candidate-scan-template.md` 형식으로 남긴다.
이 리포트는 발행 문서가 아니라, 어떤 notes를 왜 후보로 보았는지 남기는 작업 기록이다.

후보 판단 상태는 아래 값을 사용한다.

- `draft-now`: 지금 초안을 만들어도 되는 후보
- `series-candidate`: 단편보다 시리즈 흐름으로 볼 가능성이 큰 후보
- `hold`: 지금은 보류하고 이후 누적을 더 보는 후보
- `needs-more-notes`: 블로그 초안으로 만들기 전에 notes 보강이 필요한 후보

### 1차 필터

1차 필터에서는 문서 본문 전체를 읽기 전에 아래 정보로 후보를 좁힌다.

- 디렉토리
- 파일명
- 최근 수정일
- 파일 길이
- 최근 생성 여부

예를 들어 `notes/java` 아래 문서가 많아졌다면,
우선 최근 며칠 사이 수정된 문서와 새로 추가된 문서를 본다.
그다음 파일명 접두사와 주제 축을 기준으로 관련 문서를 묶는다.

### 주제 묶음 규칙

같은 디렉토리 안의 문서는 파일명만으로도 어느 정도 주제 축이 보일 수 있다.

예:

- `java-synchronized.md`
- `java-volatile.md`
- `java-locksupport-reentrantlock.md`
- `java-producer-consumer-wait-notify.md`

이런 문서들은 `java 동시성`이라는 상위 주제로 묶을 수 있다.

파이프라인은 먼저 개별 문서를 즉시 발행 후보로 보기보다,
이런 상위 묶음이 형성되는지부터 판단할 수 있어야 한다.

즉, `notes/java`에 문서가 많이 쌓였다는 사실 자체가 곧바로 전부 읽어야 한다는 뜻은 아니고,
`어떤 축으로 쌓이고 있는가`를 먼저 본다는 뜻이다.

### 최근성 우선 원칙

새로운 초안을 만들 때는 아래 순서로 우선순위를 둔다.

1. 최근 생성되거나 수정된 문서
2. 최근 문서와 직접 연결되는 같은 주제 문서
3. 시리즈 후보를 이루는 인접 문서
4. 오래됐지만 아직 승격되지 않은 문서

이 규칙이 있으면 최근 학습 흐름을 중심으로 블로그 후보를 추적할 수 있고,
이미 오래전에 닫힌 문서를 매번 다시 스캔하는 비용을 줄일 수 있다.

### 추가 독해 기준

1차 필터로 후보를 좁힌 뒤에는 필요한 문서만 추가로 읽는다.

추가로 읽는 기준은 아래와 같다.

- 최근 문서가 이전 문서의 연장선처럼 보이는가
- 제목만 봐도 같은 시리즈 축으로 연결되는가
- 단편으로 쓰기엔 맥락이 비는가
- 설명 중복 없이 하나의 글로 묶을 수 있는가

즉, 관련 문서를 읽는 행위도 무조건 전부 읽기가 아니라,
초안 판단에 필요한 만큼만 확장하는 방식으로 간다.

## 기존 축과 신규 축의 판단

문서가 누적되면 아래 두 경우를 구분해야 한다.

- 기존에 이미 형성된 학습 축 위에 새 문서가 덧붙는 경우
- 전혀 새로운 축의 문서가 들어오는 경우

예를 들어 `notes/java`에서 동시성 문서가 계속 추가된다면,
새 문서는 독립된 단편일 수도 있지만 기존 `java 동시성` 묶음의 일부일 수도 있다.

이때 파이프라인은 아래를 함께 본다.

- 새 문서 하나만으로 독립성이 충분한가
- 이전 문서를 함께 읽어야 더 자연스러운가
- 시리즈형 묶음으로 발전할 가능성이 있는가

## 중복 방지 원칙

기존 블로그에 이미 발행된 주제와 너무 겹치는 경우에는,
새 초안을 만들기 전에 아래를 먼저 확인한다.

- 같은 주제를 더 좋은 구조로 다시 쓰는 것인가
- 기존 글의 후속편인가
- 같은 내용을 notes 표현만 바꿔 다시 옮기는 것인가

즉, 초안 생성은 `notes에 있다`는 이유만으로 바로 하지 않고,
기존 블로그와의 관계까지 보고 결정한다.

## 초안 생성 단위

초안은 기본적으로 `한 notes 문서 -> 한 blog draft`를 기준으로 시작한다.

다만 아래 경우에는 예외가 가능하다.

- 여러 notes 문서가 하나의 상위 주제로 자연스럽게 묶이는 경우
- 하나의 notes 문서가 너무 커서 블로그 글 한 편으로 읽기 어려운 경우

즉, 초안 생성 시점에 아래 세 가지 선택이 가능하다.

- `1 notes -> 1 draft`
- `N notes -> 1 draft`
- `1 notes -> N drafts`

기본값은 가장 단순한 `1 notes -> 1 draft`다.

## 시리즈와 단편 판단 시점

시리즈 여부는 초안 생성 시점 또는 초안 검토 시점에 판단한다.
학습 중간에는 미리 고정하지 않는다.

### 단편으로 보는 기준

- 하나의 글로 독립적으로 읽힌다.
- 앞 글을 읽지 않아도 이해 가능하다.
- 주제 범위가 한 편 안에서 자연스럽게 닫힌다.

### 시리즈로 보는 기준

- 같은 상위 주제 아래 문서가 여러 개 연속으로 쌓였다.
- 한 글 안에 담기엔 범위가 너무 넓다.
- 자연스러운 순서가 존재한다.
- 시리즈 번호를 붙였을 때 독자가 흐름을 더 잘 이해한다.

기본값은 단편이다.
시리즈는 분명한 연결성이 있을 때만 선택한다.

## 초안 생성 결과물

블로그 초안은 최소한 아래 요소를 포함해야 한다.

- 제목 초안
- `description` 초안
- `categories` 초안
- `tags` 초안
- 도입
- 본문 주요 섹션
- 마무리
- source notes 정보

초안은 notes 원문을 그대로 복사한 상태가 아니라, 블로그 독자 기준으로 재구성된 상태여야 한다.

여기서 `source notes 정보`는 최소한 아래를 뜻한다.

- 어떤 notes 문서에서 출발했는가
- 단일 문서 기반인지, 여러 문서 묶음인지
- 시리즈 후보인지 단편 후보인지

이 정보는 초안 상단 메모나 내부 메타데이터 형태로 남길 수 있다.

초안을 새로 만들 때는 `blog-drafts/_templates/draft-template.md`를 기준으로 시작한다.
초안은 처음에는 `blog-drafts/inbox/`에 둔다.

템플릿의 주석 메타데이터는 발행용 frontmatter가 아니라 내부 추적용이다.
실제 블로그 소스 저장소로 옮길 때는 필요한 경우 이 내부 메타데이터를 제거하고,
블로그가 사용하는 frontmatter만 남긴다.

## 검토 포인트

초안 검토 시 아래 항목을 확인한다.

- 제목이 글의 주제를 정확히 드러내는가
- 설명문이 범위와 관점을 잘 요약하는가
- 도입부가 왜 이 글을 읽어야 하는지 보여주는가
- 본문 구조가 흐름 중심으로 읽히는가
- 불필요한 개인 메모나 중복 설명이 남아 있지 않은가
- 마무리가 핵심을 다시 닫아주는가
- 시리즈와 단편 중 어느 쪽이 더 자연스러운가

## 승인 흐름

초안은 아래 흐름으로 다룬다.

1. 초안 생성 후 `blog-drafts/inbox/`에 둔다.
2. 검토를 시작하면 `blog-drafts/review/`로 이동한다.
3. 승인된 초안은 `blog-drafts/approved/`로 이동한다.
4. 실제 발행이 끝나면 `blog-drafts/published/`로 이동해 보관한다.

여기서 중요한 점은 `초안 생성`과 `발행 승인`을 분리하는 것이다.
초안은 여러 번 생성, 갱신될 수 있지만, 발행 판단은 별도로 한다.

## 자동화의 역할

자동화는 아래 역할까지만 맡는다.

- 블로그 초안 후보 감지
- notes 기반 초안 생성 또는 갱신
- 제목/설명문/카테고리/태그 초안 제안
- 시리즈형 제목안과 단편형 제목안 제안

자동화가 맡지 않는 일은 아래와 같다.

- 무조건적 발행 결정
- 기존 글 일괄 수정
- 사람 검토 없는 공개 반영

## 기존 글과 신규 글의 분리

- 기존 블로그 글은 이 파이프라인에 소급 적용하지 않는다.
- 신규로 승격되는 글부터 이 파이프라인을 적용한다.
- 기존 글을 손볼지 여부는 별도 ROI 판단으로 처리한다.

즉, 이 파이프라인은 앞으로 들어오는 글을 위한 흐름이다.

## 파일명과 구조에 대한 기본 원칙

초안 파일명은 당장 블로그 발행 슬러그와 완전히 일치할 필요는 없다.
처음에는 사람이 검토하기 쉬운 이름이면 충분하다.

다만 최종적으로는 아래 정보가 드러나는 편이 좋다.

- 주제
- 시리즈 여부
- 시리즈라면 순서

예:

- `mysql-architecture-buffer-pool.md`
- `mysql-architecture-4-buffer-pool.md`
- `rearchitecturing-to-reduce-cycles-and-duplicate-code.md`

상태는 파일명 접미사보다 디렉토리 이동으로 표현하는 것을 우선한다.

후보 스캔 리포트는 초안과 구분하기 위해 파일명에 `scan`을 포함한다.

예:

- `java-concurrency-scan.md`
- `database-index-scan.md`

## 향후 확장 포인트

이 문서는 이후 아래 항목으로 확장할 수 있다.

- 승인 완료 초안의 이동 방식
- 블로그 소스 저장소로 넘기는 절차
- 실제 하네스 규칙

현재 단계에서는 별도 메타데이터 상태값보다, 디렉토리 기반 상태 관리가 더 단순하고 실용적이라고 본다.
