# Java `volatile`

## 개요

`volatile`은 여러 스레드가 공유하는 변수의 값을 읽고 쓸 때,
한 스레드의 변경 결과가 다른 스레드에서 보이도록 가시성을 보장하는 키워드다.

멀티스레드 환경에서는 분명히 값을 변경했는데도 다른 스레드가 이전 값을 계속 읽을 수 있다.
이 문서는 이런 메모리 가시성 문제와 `volatile`의 역할을 간단히 정리한다.

## 왜 필요한가

멀티스레드 환경에서는 여러 스레드가 같은 공유 변수에 접근할 수 있다.
이때 한 스레드가 값을 바꿨다고 해서 다른 스레드가 그 값을 즉시 본다고 가정하면 안 된다.

이유는 CPU 캐시, 레지스터, 컴파일러 최적화 같은 요소 때문에
각 스레드가 값을 읽고 쓰는 방식이 항상 단순하지 않기 때문이다.

결과적으로 한 스레드가 `flag = true`로 값을 바꿨더라도,
다른 스레드는 계속 `false`를 읽을 수 있다.

## 메모리 가시성

메모리 가시성은 한 스레드가 변경한 값을 다른 스레드가 확인할 수 있는지를 의미한다.

가시성이 보장되지 않으면 다음과 같은 문제가 생길 수 있다.

- 한 스레드는 분명히 값을 변경했다.
- 다른 스레드는 그 변경을 바로 보지 못한다.
- 그래서 종료되어야 할 반복문이 계속 돌거나, 잘못된 분기 처리가 발생할 수 있다.

즉, 문제의 핵심은 값이 바뀌었는지 자체보다,
그 변경이 다른 스레드에 언제 보이는지 보장되지 않는다는 점이다.

## `volatile`의 역할

`volatile`을 붙인 변수는 한 스레드의 쓰기 결과가 다른 스레드의 읽기에서 보이도록 보장한다.

예를 들어 한 스레드가 `volatile` 변수의 값을 변경하면,
다른 스레드는 그 변수를 읽을 때 최신 값을 볼 수 있다.

즉, `volatile`은 여러 스레드가 공유하는 값에 대해 메모리 가시성을 보장하는 데 사용한다.

## `happens-before`와의 관계

자바 메모리 모델에서는 `happens-before` 관계가 성립해야
한 스레드의 작업 결과가 다른 스레드에서 보이는 것이 보장된다.

`volatile` 변수에 대한 쓰기와 이후의 읽기 사이에는 이런 보장이 생긴다.
그래서 `volatile`은 단순한 문법 요소가 아니라,
멀티스레드 환경에서 가시성을 보장하기 위한 메모리 모델 규칙과 연결된다.

## `volatile`이 해결하지 못하는 것

`volatile`은 가시성은 보장하지만 원자성은 보장하지 않는다.

예를 들어 `count++`는 한 번의 동작처럼 보이지만 실제로는
읽기, 증가, 쓰기의 여러 단계로 나뉜다.
그래서 여러 스레드가 동시에 수행하면 값이 꼬일 수 있다.

이런 경우에는 `volatile`만으로는 부족하고,
`synchronized`, `Lock`, `AtomicInteger` 같은 방법이 필요하다.

## 성능

`volatile`은 일반 변수보다 비용이 더 들 수 있다.

다만 이것을 단순히 "CPU 캐시를 아예 사용하지 않는다"로 이해하면 정확하지 않다.
정확히는 가시성과 읽기/쓰기 순서를 보장하기 위한 추가 제약이 생긴다고 보는 편이 맞다.

즉, `volatile`은 필요할 때 사용하는 것이 맞지만,
모든 공유 변수에 습관적으로 붙이는 키워드로 이해하면 곤란하다.

## 핵심 정리

- `volatile`은 여러 스레드가 공유하는 변수의 가시성을 보장한다.
- 한 스레드의 변경 값을 다른 스레드가 볼 수 있게 한다.
- `volatile` 변수의 쓰기와 읽기 사이에는 `happens-before` 관계가 성립한다.
- `volatile`은 원자성까지 보장하지는 않는다.
- `count++` 같은 복합 연산에는 `synchronized`나 원자 클래스가 필요하다.
