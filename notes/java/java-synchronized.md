# Java `synchronized`

## 개요

`synchronized`는 여러 스레드가 같은 공유 자원에 동시에 접근할 때 생길 수 있는 문제를 막기 위한 자바의 기본 동기화 방법이다.
특히 출금, 재고 차감, 카운트 증가처럼 여러 단계로 이루어진 작업을 안전하게 보호할 때 자주 등장한다.

이 문서는 동시성 문제와 임계 영역이 무엇인지부터 시작해서,
`synchronized`가 어떤 락을 기준으로 동작하는지, 그리고 `volatile`과 무엇이 다른지까지 정리한다.

## 왜 필요한가

여러 스레드는 같은 객체나 같은 변수에 함께 접근할 수 있다.
문제는 어떤 작업이 한 줄처럼 보여도 실제로는 여러 단계로 나뉘는 경우가 많다는 점이다.

예를 들어 출금 로직을 생각해보자.

1. 현재 잔액을 읽는다.
2. 출금 가능한지 확인한다.
3. 잔액에서 금액을 뺀다.
4. 변경된 잔액을 다시 저장한다.

이 과정 중간에 다른 스레드도 같은 계좌에 접근하면,
둘 다 같은 잔액을 기준으로 계산해서 잘못된 결과가 나올 수 있다.
이런 식으로 여러 스레드의 실행이 섞이면서 데이터가 깨지는 문제가 동시성 문제다.

## 동시성 문제와 임계 영역

동시성 문제는 여러 스레드가 공유 자원에 동시에 접근하면서,
실행 순서에 따라 결과가 달라지거나 잘못되는 문제를 말한다.

여기서 공유 자원은 여러 스레드가 함께 사용하는 객체 상태, 필드, 컬렉션, 계좌 잔액 같은 값이다.

임계 영역은 이런 공유 자원에 접근하는 코드 중에서,
한 번에 하나의 스레드만 실행해야 안전한 구간을 말한다.

즉, 임계 영역은 "동시성 문제가 발생할 수 있는 구간"이라고 이해해도 된다.
다만 더 정확히 말하면, 공유 자원에 대한 접근이 들어 있어서 동시 실행을 허용하면 안 되는 구간이다.

출금 예제에서는 잔액 확인부터 차감 후 저장까지가 하나의 임계 영역이 된다.
이 구간이 쪼개져서 동시에 실행되면 잔액이 꼬일 수 있기 때문이다.

## `synchronized`의 동작 방식

`synchronized`는 특정 객체의 락을 기준으로 임계 영역에 한 번에 하나의 스레드만 들어오게 만든다.

자바의 모든 객체는 모니터 락이라고 부르는 내부 락과 연결되어 있다고 이해하면 된다.
스레드가 `synchronized` 메서드나 블록에 들어가려면 먼저 그 락을 획득해야 한다.

이미 다른 스레드가 같은 락을 가지고 있으면,
나머지 스레드는 락이 풀릴 때까지 기다린다.
그리고 메서드 실행이 끝나거나 블록을 빠져나오면 락을 반납한다.

중요한 점은 "`이 인스턴스 전체에 한 번에 하나의 스레드만 접근 가능하다`"가 아니라,
"같은 락을 기준으로 보호되는 코드에는 한 번에 하나의 스레드만 들어갈 수 있다"는 것이다.

즉, 서로 다른 객체를 잠그면 동시에 실행될 수 있다.
반대로 같은 객체를 기준으로 잠그면 그 객체에 대한 임계 영역은 직렬화된다.

## 메서드와 블록

인스턴스 메서드에 `synchronized`를 붙이면 그 메서드는 현재 객체인 `this`를 락으로 사용한다.

```java
public synchronized void withdraw(int amount) {
    if (balance >= amount) {
        balance -= amount;
    }
}
```

위 메서드는 같은 계좌 객체에 대해 동시에 실행되지 않는다.
하지만 서로 다른 계좌 객체라면 각각 다른 락을 사용하므로 동시에 실행될 수 있다.

특정 구간만 잠그고 싶으면 `synchronized` 블록을 사용할 수 있다.

```java
public void withdraw(int amount) {
    synchronized (this) {
        if (balance >= amount) {
            balance -= amount;
        }
    }
}
```

블록 방식은 메서드 전체가 아니라 정말 필요한 구간만 임계 영역으로 묶을 수 있다는 점에서 더 유연하다.

## 출금 예제로 보면

출금 문제의 핵심은 `balance -= amount` 한 줄만 보호하면 끝나는 것이 아니라는 점이다.
대개는 잔액 확인과 차감이 하나의 의미 있는 작업이기 때문에 함께 보호해야 한다.

```java
public synchronized boolean withdraw(int amount) {
    if (balance < amount) {
        return false;
    }

    balance -= amount;
    return true;
}
```

이렇게 하면 한 스레드가 출금 검증과 차감을 끝낼 때까지
다른 스레드는 같은 계좌 객체의 출금 로직 안으로 들어오지 못한다.

결국 "`검사하고 변경하는 과정`을 하나의 원자적인 흐름처럼 묶어준다"는 점이 중요하다.
완전히 수학적인 의미의 원자 연산이 되는 것은 아니지만,
적어도 같은 락을 공유하는 스레드들 사이에서는 중간에 끼어들 수 없게 만든다.

## `volatile`과 무엇이 다른가

`volatile`은 값을 읽고 쓸 때 가시성을 보장하는 키워드다.
즉, 한 스레드의 변경 결과를 다른 스레드가 볼 수 있게 해준다.

반면 `synchronized`는 가시성뿐 아니라 상호 배제도 제공한다.
즉, 최신 값이 보이게 할 뿐 아니라 동시에 들어오는 것 자체를 막는다.

그래서 `synchronized` 블록이나 메서드 안에서 일어난 변경은
락 획득과 반납을 기준으로 메모리 가시성도 함께 보장된다.
이 때문에 이런 구간에서는 별도로 `volatile`을 붙이지 않아도 되는 경우가 많다.

하지만 의미가 완전히 같은 것은 아니다.

- `volatile`: 값의 변경을 다른 스레드가 볼 수 있게 한다.
- `synchronized`: 한 번에 하나의 스레드만 들어오게 하고, 그 과정에서 가시성도 보장한다.

즉, 출금처럼 읽기와 검증과 쓰기가 함께 묶여야 하는 문제는 `volatile`만으로 해결되지 않는다.

## 주의할 점

`synchronized`가 붙었다고 해서 모든 동시성 문제가 자동으로 해결되는 것은 아니다.
어떤 공유 자원을 어떤 락으로 보호할지 일관되게 정해야 한다.

예를 들어 같은 `balance`를 수정하면서 어떤 코드는 `this`로 잠그고,
다른 코드는 전혀 잠그지 않거나 다른 객체로 잠그면 보호가 깨진다.

또한 락 경쟁이 많아지면 성능이 떨어질 수 있다.
그래서 임계 영역은 가능한 한 짧고 명확하게 유지하는 편이 좋다.

## 핵심 정리

- 동시성 문제는 여러 스레드가 공유 자원에 동시에 접근하면서 결과가 꼬이는 문제다.
- 임계 영역은 공유 자원에 접근하는 코드 중 한 번에 하나의 스레드만 실행해야 하는 구간이다.
- `synchronized`는 특정 객체의 모니터 락을 기준으로 임계 영역을 보호한다.
- 인스턴스 `synchronized` 메서드는 보통 `this`를 락으로 사용한다.
- 같은 락을 공유하는 코드에는 한 번에 하나의 스레드만 들어갈 수 있다.
- `synchronized`는 상호 배제와 메모리 가시성을 함께 제공한다.
- `volatile`은 가시성만 보장하므로 출금처럼 복합 연산이 있는 경우에는 충분하지 않다.
